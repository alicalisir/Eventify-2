# Persona Classification Pipeline — Final Report

End-to-end CatBoost persona classifier trained on persona-conditioned synthetic
smartphone telemetry generated by `FakeData/simulation/` and consumed by the
ML pipeline in `MLPipeline/`.

## 3. Train / validation / test methodology

| Split | Source | Selection | Size |
|---|---|---|---|
| **Train** | `outputs/train_raw/` simulation | 85 % of training users, stratified by persona | 850 |
| **Val**   | same simulation | 15 % held out by user-id (no leak across rows because features are per-user) | 150 |
| **Test**  | `outputs/test_raw/` *separate* simulation | Different RNG seed (`RANDOM_SEED_TEST = 1337`) and disjoint user_ids (offset +1 000 000); **never seen during training or hyperparameter tuning** | 300 |

Key guarantees:

- **No user leakage.** Per-user aggregation means each user is exactly one row;
  splitting on rows == splitting on users.
- **Test ≠ train population.** The held-out test isn't just held-out users from
  the same simulation — it is an *entirely separate simulation* with a
  different seed. This stress-tests generalization to unseen agents while
  staying inside the synthetic distribution.
- **Stratified by persona** in both the train/val split and the test
  generation, so every persona is represented.
- **Same feature pipeline** is used for train, val, test, and (eventually)
  real-world inference: `feature_engineering.features_from_directory()`.
  This guarantees no train/inference skew.

## 6. Strengths

1. **End-to-end reproducibility.** Two seeds (`42` for train, `1337` for test)
   and a frozen simulator deterministically reproduce the entire dataset.
2. **No train/inference skew.** `feature_engineering.features_from_directory`
   is the single feature path. Training reads training CSVs through it; future
   real-data inference will read real CSVs through the same function.
3. **Honest held-out evaluation.** Test users come from a separate simulation
   (different seed, different agents, disjoint IDs) — not just held-out rows
   of the training simulation.
4. **Class imbalance handled twice:** balanced sampling in the simulator +
   `auto_class_weights="Balanced"` in CatBoost.
5. **Interpretable.** CatBoost native gain + SHAP agree on the same top
   features, which match domain expectations (Oyuncu → gaming category, etc).
6. **Soft outputs available.** `predict_proba` gives a probability over all
   12 personas — needed for HIBRIT-style soft assignment downstream.

## 7. Synthetic-to-real limitations

These are the gaps to expect when the same model meets real device telemetry:

| Limitation | Why it matters | Mitigation |
|---|---|---|
| **Distribution gap.** The simulator uses Istanbul-shaped POIs and a fixed app catalog. Real telemetry has different mobility geometry, OS-level battery saver gaps, and a long-tail of apps not in `config.APPS`. | Models trained purely on synthetic data tend to over-rely on features that are systematically different in real data (e.g. `share_cat_short_video` magnitude). | Treat synthetic training as **pre-training**, then fine-tune on a small labelled real subset (self-report survey or weak-label bootstrap). The pipeline already supports this — just point `TRAIN_RAW_DIR` at real CSVs. |
| **App-name brittleness.** New apps appear constantly. | Direct app-name features become stale. | Features are already **category-level**, not app-name-level — much more robust. The category map will need updates yearly, not weekly. |
| **Missing-data realism.** Real users have offline gaps, permission denials, multi-device usage. The simulator emits dense events. | Real feature distributions for `total_screen_min`, `unlocks_total` will be lower and more long-tailed. | Add missing-data injection to the simulator (drop random user-days), and use rank/share features (already done) over raw counts. |
| **HIBRIT is theory-only.** Real "hybrid" users aren't necessarily Dirichlet blends; they're often time-varying. | Recall on HIBRIT is genuinely hard (0.58 here). | Consider modeling persona as a *time-varying* target (one persona per week) instead of a single user-level label, or treat HIBRIT as an explicit "low-confidence / mixed" bucket. |
| **The simulator generates the labels.** Strong recoverability (94 %) partly reflects the simulator's *internal* consistency between traits and behavior. | Real labels (self-reported personas) carry noise that the synthetic data doesn't. | Expect a 10-20 pp accuracy drop on real data even with good domain transfer. |
| **Geography is one city.** | Region-specific features (transit speeds, POI density) won't transfer. | Re-cache POIs per deployment region. The simulator is region-pluggable. |

## 9. How to use the trained model

```python
from catboost import CatBoostClassifier
import json, pandas as pd
from MLPipeline import feature_engineering as fe

# 1. Load model
model = CatBoostClassifier()
model.load_model("MLPipeline/outputs/model/catboost_persona.cbm")
classes = json.load(open("MLPipeline/outputs/model/label_encoder.json"))["classes"]
feature_cols = json.load(open("MLPipeline/outputs/model/feature_columns.json"))

# 2. Compute features for ANY directory of raw telemetry CSVs
#    (gps_pings.csv, app_sessions.csv, screen_events.csv [, episode_log.csv])
feats = fe.features_from_directory("path/to/raw_csvs", include_episode_features=False)
X = feats[feature_cols].to_numpy(dtype=float)

# 3. Predict
proba = model.predict_proba(X)              # (N, 12)
top1  = [classes[i] for i in proba.argmax(axis=1)]
```

The same `feature_engineering` module that produced the training features
will produce inference features — so there is no skew.
